Concept 1

The chain is the unit of value — not the model

Business objective  →  Use case      →  Persona          →  Scenario chain        →  KPI moved
"cut HR ticket load"   policy Q&A        Raj (employee)      Assess→Classify→          ticket deflection
                                          Sofia (HR Ops)      Quantify→Decide→          0% → 55%
                                                              ★HITL(16)→Act→Learn

Memorize the five slots. An AI service answering a question is a demo; a scenario chain is a production system: agents in named roles, on governed data, gated by a human where it matters, leaving evidence at every step, moving a KPI a named business persona owns.

The test you'll be held to all day (and in the capstone): if you cannot fill in all five slots for a scenario, it is not ready to build.

Concept 2

The canonical 24-step chain and the six-role runtime

Every scenario specializes the same 24-step backbone: W1 foundation (1–10, the data/identity/evidence planes built once), W2 pilot (11–18, what you run today), W3 scale & fuse (19–24). Exactly one enforceable human gate, always at step 16.

ASSESS(13) → CLASSIFY(14) → QUANTIFY(15) → DECIDE(16) → ★ HITL gate(16) → ACT(17) → LEARN(23)
  • Gate modes: ZERO_TOUCH auto-approved, fully evidenced · ACK_ONLY a human confirms, doesn't redesign · ESCALATION auto above confidence 0.70, else routed to a human with context · HITL hard gate, a named human approves.
  • Evidence: every executed step writes one 14-field ledger row. If a step didn't write a row, it didn't happen.
  • ACT is structurally unreachable without an approved gate — the runtime enforces it, not the scenario author. Read it yourself: the whole engine is ~200 lines.

Concept 4

How it works in production — six layers

Sources (Workday, SuccessFactors, ServiceNow) feed the OneLake medallion (Bronze raw → Silver canonical/PII-tokenized → Gold agent-safe views); the KB pipeline embeds into Azure AI Search; the Container Apps runtime (API + MAF ChainRunner + MCP tools + gate) executes chains, writing every step to the PostgreSQL ledger; Entra/Purview/Monitor govern; employees consume via Copilot, the dashboard, Adaptive Cards, and Power BI. Your laptop's data/ folder plays the Gold layer — you've been using this architecture all morning.

Concept 5

Grounding: the knowledge base and vector search

  • 30 synthetic documents in data/kb/ — policies, guides, FAQs, localized (es/de). Frontmatter metadata on every doc, including scenarios: — which chains this document grounds. A test fails if any scenario loses coverage.
  • Chunked at sections → 132 records, each with an embedded content_vector column.
  • Offline vector search works today: a deterministic 256-dim feature-hash embedding — real cosine similarity, zero credentials. Live mode swaps in text-embedding-3-small (1536-dim) and Azure AI Search hybrid retrieval (BM25 + kNN over an HNSW index) — same code path.
  • Design gap to notice: garnishment topics are deliberately NOT in the KB — that's why hr-hrsd-08 escalates. Coverage is a governance decision, not an accident.

Lab 4 · The knowledge base

Search it three ways, then build the vector index (30 min)

academy kb stats                                  # corpus + per-scenario coverage table
academy kb search "time away when my mother is ill"
academy kb search "time away when my mother is ill" --vector
academy kb build --vectors                        # writes chunks.jsonl with content_vector

Tasks:

  1. Compare the keyword and vector rankings for the query above. The mock embedding is soft-lexical (feature-hashed) — explain why its ranking differs from keyword's, and what a live semantic embedding would do differently.
  2. Open data/kb/index/chunks.jsonl and inspect one record: find the metadata fields and the 256 floats of its content_vector.
  3. Stretch (needs an Azure AI Search service): set AZURE_SEARCH_* in .env and run academy kb push — it creates the HNSW index and uploads all 132 chunks.
kb stats shows 30 documents, 132 chunks, and all nine scenarios covered. Your jsonl record has id, doc_id, title, section, content, scenarios, content_vector.

Concept 6

Integrating with M365 Copilot: agent → plugin → your API

M365 Copilot chat (Teams)
  └─ Declarative agent "HR Service Delivery"      declarativeAgent.json (schema v1.7)
       └─ API plugin action (OpenAPI)             ai-plugin.json (v2.4) + openapi.json
            └─ academy-api                        the REST contract from Lab 6
                 └─ ChainRunner → the 9 chains    unchanged: gate at 16, ledger, KPIs
  • The declarative agent's instructions route employee asks to scenario ids, forbid answering HR policy from model knowledge, and require honest gate/escalation reporting — read them, they're the governance layer in prose.
  • The plugin manifest's functions must match the API's operationIds — a repo test (tests/test_m365_package.py) enforces it.
  • Dual-tenant lifecycle: test on the Deloitte tenant (synthetic data, X-Api-Key) → deploy at a customer (their Azure, Entra OAuth). Only the stamped API base URL and the plugin's auth block change.

Lab 7 · Ship it to Copilot

Package, sideload, and test your agent (45 min · needs tenant access)

# 1 · generate the package icons (once):
python scripts/make_icons.py

# 2 · build the app package against the class API URL (instructor provides it),
#     or your own deployment from infra/deploy.ps1:
m365/package.ps1 -ApiBaseUrl https://<the-deployed-api-url>

# 3 · sideload m365/dist/hr-service-delivery-agent.zip:
#     Teams → Apps → Manage your apps → Upload an app → Upload a custom app

Test protocol — run these four conversation starters and verify:

  1. Policy Q&A — the answer keeps the [source: …] citation and Raj's US-CA rider.
  2. Get a letter — the agent relays the letter and the ACK_ONLY framing.
  3. Who do I ask? (garnishment) — the agent reports the escalation to Payroll Operations as routing-with-context, not failure.
  4. En español — the answer comes back in Spanish.

Then the trust tests: ask an HR question without naming a scenario (instructions must route it), and ask something the KB can't answer (it must escalate, not invent).

All four starters behave; the agent never invents policy. You just integrated an HR process into an agentic orchestration surfaced in M365 Copilot — the module's goal.

Capstone

Ship scenario hr-hrsd-10 — your own HR process, end to end

Pick an uncovered Tier-0 request (holiday calendar lookup, expense-policy Q&A, org-chart "who do I ask about X", onboarding buddy FAQ…) and integrate it into the orchestration:

  1. Spec — ScenarioSpec with personas, synthetic KPIs, Azure service seats, and a HITL mode you defend in the module docstring (name the harm).
  2. Handlers — all six roles; deterministic in mock mode.
  3. Data — any new KB document under data/kb/ with correct frontmatter (including scenarios: [hr-hrsd-10]) so coverage tests see it.
  4. Brief — scenarios/hr-service-delivery/hr-hrsd-10-<name>.md in the house format.
  5. Register it — the contract tests must pass unchanged: pytest -q is your acceptance bar. Add a conversation starter to the Copilot agent and re-package (Lab 7) for full credit.

Grading: chain correctness 40% · business case (the five-slot line) 30% · code quality & tests 20% · the brief 10%.
